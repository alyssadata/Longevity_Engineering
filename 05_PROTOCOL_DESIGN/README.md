# 05_PROTOCOL_DESIGN

This section is where the repository moves from analysis into structured protocol thinking.

Not protocol performance.
Not protocol marketing.
Not protocol certainty.

Protocol design.

That distinction matters.

The purpose of this section is not to claim that aging has already been solved.
It is to define what kind of evidence, structure, and constraint would be
required before any intervention logic deserves protocol form.

## Core Position

Protocol design cannot be built on mechanism alone.

It cannot be built on biomarker enthusiasm alone.

It cannot be built on intervention novelty alone.

It has to be built where four things meet:

- biological plausibility
- measurement discipline
- functional relevance
- explicit risk boundaries

Without those, protocol design turns into theater.

## What the Repository Has Established So Far

The hallmarks section established that aging is not one process.
It is a network of interacting failures across multiple systems.

The biomarker section established that the measurement problem is structural.
No biomarker class is yet strong enough to justify confident individual-level
decisions on its own.

The intervention section established that the strongest human-grounded
interventions are not the most molecularly ambitious.
They are exercise, high-quality dietary patterning, and sleep treatment.

The notes section established the governing rule that now applies here:

When biomarkers conflict with function, function takes precedence.

That rule governs this entire section.

## Why This Section Matters

Protocol design is where false confidence becomes dangerous.

A good mechanism can be overstated.
A biomarker shift can be misread.
An intervention can sound credible long before it is ready for structured use.

This section exists to prevent that drift.

Its purpose is to answer questions such as:

- what kind of intervention is mature enough for protocol logic
- what level of biomarker support is enough to matter
- when function outweighs molecular ambiguity
- how risk should constrain design
- what belongs in a foundation layer versus an exploratory layer
- what should remain in research evaluation rather than move toward use

## Governing Rule

Function takes precedence when biomarkers conflict with it.

That means protocol design in this repository must remain anchored to:

- capacity
- resilience
- recovery
- physical performance
- physiological function
- meaningful human outcomes

A protocol that improves clocks without improving function should be treated
with skepticism.

A protocol that improves function while the biomarker story remains incomplete
may still deserve serious weight.

This does not make biomarkers irrelevant.
It means they are support layers, not sovereign layers.

## What Counts As Protocol-Relevant Evidence

For an intervention or intervention class to become protocol-relevant in this
repository, it should satisfy enough of the following:

- plausible biological mechanism
- meaningful target-hallmark logic
- interpretable biomarker structure
- functional or physiological relevance
- acceptable risk for the intended population
- manageable implementation burden
- enough human evidence to justify structured use
- enough clarity to distinguish support from speculation

No intervention needs perfect evidence.

But protocol design requires more than excitement.

## What This Section Will Not Do

This section will not:

- pretend exploratory interventions are validated
- treat molecular sophistication as superior to human function
- collapse research interest into recommendation
- hide uncertainty inside polished protocol language
- ignore population fit
- ignore adherence, recovery, or real-world burden
- promote a stack simply because each component sounds promising

If an intervention is not ready, this section should say so clearly.

## Protocol Layers

Protocol design in this repository will likely separate interventions into
different structural layers.

### 1. Foundation Layer

This is the base of any credible protocol logic.

It includes interventions with the strongest human functional support and the
lowest interpretation risk.

Current repository examples:
- exercise
- high-quality dietary patterning
- sleep and recovery treatment logic

These are not minor lifestyle notes.
They are the current foundation because they already improve the things aging
actually costs.

### 2. Support Layer

This layer includes interventions that may support the foundation, improve fit,
reduce burden, or strengthen specific bottlenecks, but are not strong enough to
stand as a primary protocol core.

Examples may eventually include:
- targeted microbiome support
- narrower metabolic support
- recovery-supportive strategies
- context-specific adjuncts

This layer is secondary by design.

### 3. Exploratory Layer

This layer includes interventions that are biologically important and may become
protocol-relevant later, but are not ready now.

Current repository examples may include:
- rapamycin and related mTOR-targeting strategies
- NAD+ restoration
- senolytics
- plasma exchange logic
- telomere-related intervention
- partial epigenetic reprogramming

This layer should remain explicitly exploratory until stronger functional,
risk, and translation evidence exists.

## Protocol Design Principles

### Principle 1 | Start from function, not from prestige

The first protocol question is not:
What is the most advanced intervention?

The first protocol question is:
What reliably improves human capacity, resilience, and recovery?

### Principle 2 | Build from the lowest interpretation-risk layers upward

Interventions with strong human functional grounding should come first.

Higher-risk, higher-ambiguity interventions should never define the core
protocol before the foundation is established.

### Principle 3 | Biomarkers are support signals, not ruling signals

Biomarkers can help track direction, burden, or mechanism.

They cannot currently justify protocol escalation by themselves.

### Principle 4 | Risk must be explicit

Every protocol layer must state:

- what is known
- what is estimated
- what is uncertain
- what the main risk boundary is
- what would count as failure
- what would count as too much burden for too little gain

### Principle 5 | Simplicity outranks elegance when function is equal

A simpler intervention structure with clearer function and lower burden should
outrank a more sophisticated structure that adds molecular complexity without
clear organismal gain.

### Principle 6 | Population fit matters

No protocol in this repository should assume that one structure fits all
populations equally.

Age, baseline burden, frailty, metabolic state, recovery capacity, and disease
context all matter.

### Principle 7 | Combination logic must earn itself

A combination belongs in protocol design only if it improves:

- function
- resilience
- bottleneck coverage
- durability
- or tolerability

If it only increases complexity, it does not belong.

## Current Starting Position

Current working position of this repository:

- protocol design should begin from behavioral and functional foundations
- protocol design should not proceed on molecular biomarker confidence alone
- exploratory interventions remain below protocol threshold until stronger
  human evidence exists
- function takes precedence when there is signal conflict
- risk and implementation burden are part of protocol quality
- foundation is not lesser because it is simpler
- technological ambition does not outrank human-grounded evidence

## Main Questions This Section Must Answer

- What belongs in a credible foundation layer?
- What evidence is enough to move something from exploratory to support?
- What level of functional improvement is enough to justify escalation?
- How should biomarkers be used without becoming sovereign?
- How should protocol design handle disagreement between molecular and
  organismal signals?
- Which combinations deserve structure, and which are just stacks?
- What should remain research-only no matter how interesting it is?
