# 05_protocol_questions

## Protocol Questions

This file tracks the protocol questions the repository has not yet resolved.

The protocol section already did major work.

It established that:

- the foundation is named and bounded
- the escalation logic is explicit
- the failure modes are named
- the governing rule is restated and applied
- protocol design must remain foundation-first, function-first, and
  risk-bounded

Those are current repository positions, not open questions here.

This file is for the protocol questions that remain unresolved **inside** that
structure.

## Core Position

The protocol problem is no longer lack of architecture.

The architecture now exists.

The real open protocol questions are about thresholds, weighting, timing, and
how the current structure should evolve without losing honesty.

That means this file is not asking:

How do we invent a protocol framework?

It is asking:

What still blocks stronger protocol confidence **within** the framework that
already exists?

## What This File Does Not Reopen

This file does **not** reopen the following current repository positions:

- protocol design cannot be built on mechanism alone
- protocol design cannot be built on biomarker enthusiasm alone
- the foundation layer currently consists of exercise, high-quality dietary
  patterning, and sleep and recovery treatment logic
- support must remain support
- exploratory importance does not equal protocol readiness
- function takes precedence when biomarkers conflict with it
- risk boundaries and nonpromotion rules are necessary

Those are current repository boundaries.

The open questions here begin after those rules.

## Main Open Protocol Questions

### 1. What counts as enough functional improvement to justify escalation?

This is one of the most important open questions in the whole protocol section.

The repository already says that function governs escalation.

What remains unresolved is threshold strength.

Questions include:

- how much improvement counts as meaningful
- how durable the improvement has to be
- whether different functional domains should carry different weights
- whether preserving function in a vulnerable person should count as strongly as
  improving function in a robust person
- when partial functional gain is enough to justify adding structure

This question matters because “function matters” is settled, but “how much
function is enough” is not.

### 2. How should different functional domains be weighted against each other?

Function is the main arbitration layer.
Function is not one thing.

The repository still needs clearer weighting logic across domains such as:

- strength
- gait and mobility
- cardiorespiratory fitness
- recovery quality
- fatigue burden
- frailty-related measures
- cognitive function where relevant
- resilience under stress

For example:

- does recovery outrank performance
- does frailty reduction outrank isolated strength gain
- when does preserving function matter more than increasing it
- how should broad organismal gain be distinguished from narrow task gain

This remains open.

### 3. What is the minimum viable protocol base before support can be added?

The repository already says foundation first.

What remains unresolved is the threshold for when the foundation is actually
real enough to support more.

Questions include:

- how structured exercise has to be before the base counts as established
- how much dietary quality has to be specified before it becomes a real layer
- how stable sleep and recovery logic has to be before escalation is honest
- whether the minimum differs by population fit

This matters because weak foundations are one of the main routes into protocol
failure.

### 4. When does a support intervention become genuinely support-worthy?

The support layer exists.
Its threshold is still not fully settled.

Open questions include:

- what degree of bottleneck clarity is enough
- what degree of functional or recovery improvement is enough
- when a support intervention is truly strengthening the base rather than only
  adding biological plausibility
- how much burden is acceptable in the support layer
- when support remains too exploratory to enter protocol structure at all

This is a major open question because the support layer is one of the easiest
places for quiet drift to occur.

### 5. When should a protocol prefer simplification over escalation?

The repository already allows de-escalation in principle.

What remains open is how aggressively simplification should be preferred when
signals are mixed.

Questions include:

- how much ambiguity is enough to justify simplification
- when rising burden should trigger removal rather than patience
- whether simplification should happen faster in frailer or lower-recovery
  populations
- how much protocol complexity is worth carrying before likely benefit falls
  behind likely burden

This is one of the main open questions for keeping the protocol architecture
clean over time.

### 6. How should protocol design handle mixed signals without collapsing into vagueness?

Mixed signals are expected.

A protocol may show:

- better function and mixed biomarkers
- better biomarkers and flat function
- better recovery but unchanged performance
- stronger adherence but weaker molecular clarity
- gains in one domain and losses in another

The repository already says function decides when there is direct conflict.

What remains open is the practical handling of more complex mixed cases.

This includes questions such as:

- when to pause
- when to hold
- when to simplify
- when to allow time for clarification
- when the ambiguity itself becomes disqualifying

### 7. What should count as a real threshold crossing from exploratory to protocol-relevant?

This is one of the most important future-facing questions in the repo.

The repository already has exploratory status and nonpromotion rules.

What it has not fully settled is what exact kind of evidence would move an
intervention upward.

Questions include:

- is one strong randomized human functional result enough
- how much biomarker-function alignment is enough
- whether threshold crossing should vary by risk tier
- whether a support-layer transition requires less evidence than a more central
  protocol role
- which interventions need qualitatively stronger evidence because their risk
  profile is structurally higher

### 8. How should sequencing change across populations?

The protocol section already says sequence matters.

What remains open is how much the sequence should shift depending on:

- frailty
- recovery capacity
- baseline function
- metabolic burden
- implementation capacity
- age and vulnerability

This matters because a correct sequence for one person may be a bad sequence
for another, and the repo still needs sharper guidance on where those branching
points begin.

### 9. How much measurement is enough before the protocol becomes measurement-heavy?

The repository already rejects biomarker governance.

What remains open is the practical question of how much measurement a protocol
should tolerate before it starts becoming:

- too burdensome
- too interpretively fragile
- too biomarker-shaped
- too hard to live
- too easy to defend rhetorically without stronger organismal gain

This is important because measurement can support clarity, but it can also
become part of burden creep.

### 10. What should the protocol do when the foundation is helping, but not enough?

This is one of the most practical open questions in the whole section.

The repository already says the foundation is strongest.

But real protocol work still needs a better answer to:

What do we do when the foundation is real, meaningful, and still insufficient
for a specific person or bottleneck?

That is the exact point where support, sequence, and nonpromotion questions all
start interacting.

## Highest-Priority Protocol Questions for the Repository

Not every protocol question carries equal structural weight.

The current highest-priority questions appear to be:

1. What counts as enough functional improvement to justify escalation?
2. What is the minimum viable protocol base before support can be added?
3. When does a support intervention become genuinely support-worthy?
4. When should simplification outrank escalation?
5. What counts as a true threshold crossing from exploratory to
   protocol-relevant?
6. How should mixed signals be handled without drifting into vagueness?

These are the protocol questions most likely to shape the next phase of the
repository.

## What Would Count As Progress on These Questions

This repository should not treat more protocol language as progress.

Progress here would mean things such as:

- clearer escalation thresholds
- clearer support-entry thresholds
- clearer simplification triggers
- better weighting across functional domains
- stronger rules for mixed-signal handling
- clearer transition criteria for exploratory interventions
- better population-specific sequencing logic

That is the kind of progress this section should track.

## Protocol Failure Modes This File Should Prevent

This file should help prevent several common failures:

### Failure Mode 1 | Endless architecture without thresholds

The protocol gets more structured, but still does not know when to move or
when not to move.

### Failure Mode 2 | Escalation bias

The protocol quietly assumes that more structure is usually better.

### Failure Mode 3 | Support drift

The support layer grows without strong enough entry rules.

### Failure Mode 4 | Simplification shame

The protocol resists removing things because reduction feels like loss instead
of correction.

### Failure Mode 5 | Mixed-signal vagueness

The protocol keeps functioning inside unresolved ambiguity without deciding
what kind of ambiguity is still acceptable.
